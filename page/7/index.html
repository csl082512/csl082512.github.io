再说小程序

发表于 2018-01-09

近日又重新开始学习小程序，说来惭愧，已经delay了一年，我现在想把它补回来。
至少三年没写代码了，虽然理解还在，但是思维和动作已经不在了。今天，我想从技术和产品上谈下小程序。
先从技术上谈谈，小程序相当于提供了一个二次开发平台，开发者可以基于庞大的微信用户生态，构建自己的想法的应用。
我特意研究了下，里面大概的文件结构是如下：
js文件-描述业务处理逻辑
wxml文件-描述布局，类似html
wxss文件-描述样式，类似css
这几个架起了最基本的模型。再基于微信app载体，制定了覆盖网络、文件、多媒体、原生服务等，允许用户开发出更加具有吸引力的应用！在有限的功能上，尽量保证能够提供给开发者比较完整的接口。我重新看了下代码，虽然只有一些简单的功能，但是已经能够保证能开发一些产品了，满足中小群体的需求。
从产品上看，这是一个开放生态，微信给了小程序很多权限，即便收敛了一些，基于微信庞大的用户，还是显得非常恐怖。比较代表性的应用是mobike，比较代表性的游戏是跳一跳。对于一些工具类，尤其是场景触发，用完即走的应用来说，再合适不过了！小程序还提供了一个位置触发的功能，基于用户的地理位置推荐估计的服务，这提供了用户打开微信的另一个理由。微信已经发展到这么庞大的阶段，如何保持新的增长力和爆点，才是关键。未来，基于这个生态，提供用户、征信、云、支付、运营、推广等服务，将会有很大的发展空间！
对比一下，我看了支付宝的小程序，确实逊色了不少，从设计到开发，差距非常明显。很早之前，我就提到，小程序可能是支付宝的一个机会，只是支付宝没有走好这步棋，在微信铺开之前，没有打出来，就几乎肯定再也没有机会了！

芝麻信用生态

发表于 2017-12-22

记录一些自己的思考：
AARON 20:31

芝麻信用的发展，生态。支付宝是一个支付工具，芝麻信用是一个生态。通过生态，各种机制，约束。商户像店铺一样入住信用生态，用户到期会收到一张信用账单，有账期，由商户提供。基于信用账单进行付款，账期由用户来付即可。

AARON 20:38

举个栗子，你在一个熟人的店里面，消费，你的脸就是你的信用，你先使用了服务。过了一周或者一个月，你要去店里面结清你的欠款，然后才能继续使用服务。超过了这个账期之后，你如果没有偿还账款，你的信用就收到影响，可能需要付违约金，服务不能使用。这是一个新的生态，连接信用和服务。支付宝，只可能成为一个最终偿还信用账期的支付工具。

AARON 20:42

将来，信用生态入住的服务，用户都可以凭借你的信用分，先使用再买单。

AARON 20:44

最开始，我的理解是，你需要搜集信息，获得数据，再构建信用。现在看来，应该是，让店铺像入住淘宝一样入住商城，自动贡献信用的数据。

AARON 20:50

区块链在其中扮演什么角色呢？用户数据都在平台上，区块链在机构间数据交换就不重要了。更多的，应该是更加深入场景，深入用户。

AARON 20:55

李丛杉认为最早在PC互联网中是以百度为中心的信息生态，连接了人和信息。随着信息生态的发展，继而是以阿里为中心的，基于消费的电商生态，连接的是人和商品。移动互联网时代，还出现了一个非常大的生态，即微信生态。“下一个就是‘信用+’的生态，要去连接的是人和服务，但是连接人和服务这个事情一直没有被打通。”李丛杉说。

她进一步指出，连接人和服务的过程中最关键的：一是资源的配置，二是服务体验，三是由于相互不见面必须先付钱再提供服务。因而下一个能够真正连接人和服务的一定是需要一个新型“信用+”的生态。

AARON 21:55

淘宝天猫生态卖的是商品，所以产生了需要担保，再发货的交易形态，支付担保模式。芝麻信用生态卖的是服务，所以只需要提供信用担保，即可享受服务，支付前置模式。

AARON 21:58

最开始是数据交换，才能拿到数据，才能计算反馈。最后，降维打击，直接让你的服务在我的平台开展，通过信用支付流水，获得用户数据。

AARON 21:59

不卖数据，不卖能力，提供生态入住服务即可。

人类组织发展

发表于 2017-12-15

在我看来，在最早的人类，和最终的人类，都是不需要中心组织的。
为什么这么认为？中心组织可以认为是中介，中介的意义是消除信息的不对称性。换句话说，什么时候不需要中介了，那就是信息绝对透明和对称的时候。
最早的人类，和蜥蜴一样，只关心三件事情：食物，危险和性。这些都是本能，最基础的东西，支撑着个体的发展。在那个时候，异常艰难，环境充满了危险，想单独获取食物，将会变得很困难。而能够很容易就获取食物的，比如狮子老虎这些，都不会有太多的担忧，于是他们不需要组织。
最早的人类，发现这样大家都没法活下去，于是形成了组织，主要的目的就是解决食物和危险问题。
最初的组织，也是很单纯的组织，也是透明的组织。捕获的食物，按照贡献分配。私下的交易，也是允许的。那个时候，食物的交易，就是一袋小麦换一袋土豆，直接的实物交易。
组织发展到后来，越来约臃肿不堪，层级关系开始出现，这是任何中心化组织发展的通病，尤其是增加了宗教。皇权与宗教，政教合一，为处于中心地位的高层，谋取了非法的利益。
人都是利益驱使的，只会为代表自己利益的阶层说话。中心组织决定的策略，永远指向一个目的，让其利益最大化。
中心机构开始登记各种信息，成为物权的最终决定者。不过我们想想，究竟谁才是物权的最终决定者？是机构组织么？我原来花了一袋小麦换来的农具，他说不是我的就不是我的了？很遗憾，确实如此！
所以问题的关键是，谁能决定你就是拥有这个东西？决定者是你，但是其他人可以帮你作证！以前的那袋土豆是你的，其它人也能证明那就是你的。因此，本质是确权。
现在的人类发展已经到了数字化时代，如何在数字化时代维护自己的利益，却显得非常困难。以前，属于我的东西，我不想给别人不能勉强，现在各种盗版，盗卖。数字化时代，我们却无法更加有利的保护我们的资产。我们的资产已经数字化了，但需要更好保护。
中介能帮助我们实现这点么？
我看未必，中介产生只是发展需要，供需双方直接发生关系才是最终的形态。因此，点对点，去中心化的直接交易，才是最终的形态。
确权和去中心化，这两点非常重要，未来人类的我发展方向应该是去中心化的，去中介的。
这两点，我在区块链技术看到了未来，只是还有很多路要走！

价值的波动

发表于 2017-11-30

人类真是一个矛盾的个体，如果一个东西的价值是波动的，那么就会吸引很多人去关注，反之，如果一个东西的价值是恒定的，则很少有人去关注它。运动是最重要的状态，只有运动的东西，才会产生波动，才会吸引人群参与。
比特币为什么这么像货币？
首先是稀缺性，然后才是波动性。稀缺意味着价值，波动意味着有人关注。如果它一直横在那里，谁会去关注？
因此，积分也一样。一层不变的积分，会让用户厌倦，只有波动起来，才会吸引更多的用户参与。同时，增加一定的神秘感和未知性，满足更多人内心的欲望！

先问一个问题：区块链和其它技术最本质的区别是什么？
这里我给一个参考，最本质的区别是确权后的数字化流通！
区块链本来就是一门基于P2P、分布式和密码学的技术，这些技术都是存在很多年了的，也没有太大的变化。因此，如果从技术上来说，它是没有区别的，只能从技术层面以上来寻找。区块链更像一种新的生产关系，解决了信任问题。信任是什么？信任最直观的表现就是信用评分。mobike为什么愿意把车给你骑？信用贷款公司为什么愿意把钱给你？完全是信任。更简单一些，你为什么愿意给朋友借钱而不是陌生人，还是信任。mobike在没有引入芝麻信用之前，做的事情就是缴纳押金，然后给予你信任度，才能使用。这里的信任度是要钱买的，你支付了你潜在违约的平均成本。
共享单车押金目前也是被大众批判的焦点，很多厂商挪用了押金，投资到新增设备上，结果就是钱没了，设备丢了，人没了。押金不应该挪用，因为那是属于用户的权益。
引入芝麻信用，能够极大缓解这个问题，因为通过分数，给厂商吃了一颗定心丸。类似，你不会借给一个陌生人钱，但如果这个陌生人是你朋友的朋友，而且还不错，你会考虑的，道理一致！
再回到信用问题上，区块链到底能做什么？
区块链能够确权，就是证明这个东西确实是属于你的，这个数据确实是你上传的，你能对你的操作行为进行信用背书，不可抵赖。既然如此，那么真实的信息搜集就有了保证，信息提供发能够对它提供的信息进行担保。这解决了信息来源的问题，基于可信的信息，后面可以建立模型，通过模型产生信用评分标准，也可以更新信息来调整评分。
信用出来后，可以提供给场景使用，同时通过场景提供数据去更新信用分数，一个反哺的闭环的过程。
信用如果仅仅认为是提供准入门槛，那就太局限了。信用也是一种支付方式，一种更加高级的支付方式。传统的支付方式是基于实物价值，表现为货币和实物担保。很简单，你和老板说，看在我的人品上，这顿饭先记账我后面再结清。你的面子支付了这顿饭，而且是实时支付了。
所以，信用支付，或许是以后支付方式的最终形态。

产品三问

发表于 2017-11-26

每个产品都需要问以下这个问题：
我的产品在什么时候帮什么用户解决什么问题？
1.什么时候？When，描述场景
2.什么用户？Who，描述用户
3.什么问题？what，描述痛点
我们详细说一下
场景分线上线下，移动端和PC端，弱场景和强场景，时效要求高和低的场景，安全性要求高和低的场景……场景往往和目的结合起来。
用户往往和场景相关，线上线下，新老用户，B端用户还是C端用户，2B还是2B2C……用户往往决定你的实现方式
痛点也结合场景用户，高频还是低频，真痛点和伪痛点……
因此，在产品设计规划阶段，就要开始从场景用户痛点三个维度去思考。
我要设计一个现场弹幕产品：

  • 我的产品在举办婚礼的现场，帮助来到现场的亲朋好友解决发表祝福的问题。
  • 我的产品在举办婚礼的现场，帮助没有来到现场的朋友，解决发表祝福的问题。

下面我们扩展思考一下，看后面是否需要返回更新我们的描述
场景是举办婚礼的现场，必须是现场么？不在现场是不是也可以，因为不一定人都会到现场去。是否可以在网络参与？时间点必须是婚礼进行中么？在婚礼前是否也可以？还有场景必须是婚礼么？其它是否也可以？比如生日宴会、年会、庆功宴等等。
用户是参加婚礼的用户，到现场和不到现场。用户的群体是什么样子？年龄层次？交互方式什么样子最合适？
痛点是缺乏表达祝福的方式。有哪些祝福？朋友之间的调侃，小辈的祝福，长辈的谆谆教导等等。祝福的表达方式是什么样子？祝福可以文字，视频，照片，语音，红包，幸运符等等。考虑微信，支付宝，还是QQ？或者自己开发一个小应用？或者H5？或者简单的短信？
这样一思考，我们发现可以有很多扩展的，我只列举了枝干部分，还有其它需要细化。
比较推荐通过思维导图进行思考，从产品开始，扩展几个要点，再扩展你想到的，再Mark下重点！

选择

发表于 2017-11-22

考虑以下几个选项，请选择：
A.新瓶装旧酒
B.老瓶装新酒
C.新瓶装新酒
D.老瓶装老酒
选择哪些？
A表示你花了很多精力去包装，但是实际上不是什么新东西。
B表示你懒得花时间精力去包装，虽然你做的实际上是新东西
C表示你花了很多时间精力去包装，东西也是新的
D表示你没有花时间精力去包装，东西依旧还是老东西
D代表的是不求上进，不求发展。
B代表的是内部改变，但外界并不知道
A代表的是对外表现积极，但是实际半吊子
C代表的是对外表现积极，实际行动也不错
可以结合做人和做事，两个来看，你到底更愿意选什么！
